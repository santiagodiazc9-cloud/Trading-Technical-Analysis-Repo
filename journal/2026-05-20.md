### What happened to the approval
The Discord approve button mis-fired at 13:28Z: the bot flagged the *expired NVDA setup* as `Approved: YES` instead of GOOGL (bot commit `aabc6b4`). A follow-up session caught it and pushed `1a406a5` ("fix(approval): correct GOOGL-2026-05-20 approval"), which moved the flag to the right block and restored the NVDA note. This cloud routine started one commit stale (`aabc6b4`), rebased onto `1a406a5`, and ran against the corrected state: GOOGL is genuinely `Approved: YES`.

Root cause of the misfire — two code bugs (flagged for Santiago, not fixed here since `scripts/` is out of routine scope):
- `discord_bot.py:134` — `update_open_positions` regex `(### .*SYMBOL.*?)(\n##|\Z)` is greedy + `re.DOTALL`, so it matches from the first `### ` heading (expired NVDA setup) and replaces the first `- Approved:` line it finds.
- `setup_validator.py parse_pending_setups` — reads only the first of the file's two `## Pending Setups` headings. I removed the duplicate heading as a data fix; `setup_validator` now resolves the real GOOGL block (`approved: true, valid: true`).

### Lesson
The approval gate is only as reliable as the bot writing to it: a fuzzy symbol-matching regex silently flagged the wrong setup. Approval writes should anchor on the exact `setup_id` (e.g. the `setup-data:json` block), never a loose symbol match — worth an ADR. Separately: an "approved" setup is not an "execute now" setup — the execution note is part of what was approved, and the routine must honor per-setup timing instructions over its generic "place order" step.
